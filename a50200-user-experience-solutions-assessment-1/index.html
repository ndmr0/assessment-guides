Plain English Roadmap

Choose

Pick a real product or project context that you understand well enough to discuss honestly.

Observe

Write what users currently experience. Avoid guessing feelings without evidence.

Ask

Collect responses from real people. Do not invent research data.

Connect

Use the same problem, users, and findings across your answers so the assessment feels like one project.

Justify

Explain why each decision helps users, not only what you personally prefer.

Academic Integrity

Permitted AI Use

AI can support

  • Clarifying UX concepts and task requirements.
  • Brainstorming possible issues, methods, and question formats.
  • Improving grammar, formatting, and structure.
  • Creating templates that students adapt to their own case study.

AI cannot replace

  • Final assessment answers.
  • Real participant responses.
  • Student-led analysis, insights, personas, or UX decisions.
  • Context-specific justification based on the student’s own project.

Student rule of thumb: AI can help you prepare and polish. Your selected product, research data, findings, persona, recommendations, and decisions must come from you.

T1 Q1

Applying UX Principles and Values

Define each principle, judge whether it succeeds or fails in your product, recommend an improvement, and justify the user impact.

Usefulness

Does the product help users achieve a real goal?

Look for: task relevance, missing features, unnecessary steps.

Aesthetics

Does the interface look clear, consistent, and appropriate?

Look for: hierarchy, spacing, colour, readability.

Desirability

Does it feel trustworthy, appealing, and worth returning to?

Look for: confidence, motivation, tone, emotional response.

Usability

Can users complete tasks easily, efficiently, and with few errors?

Look for: navigation, labels, forms, feedback, accessibility.

Function

Do features work reliably and match user expectations?

Look for: broken flows, loading, errors, performance.

Beginner definition helper

Usefulness Does this product solve the right problem for the user?
Aesthetics Does the visual design make the experience clearer and easier to trust?
Desirability Would users want to use it again, recommend it, or feel positive about it?
Usability Can users complete the task without confusion, errors, or unnecessary effort?
Function Do the features actually work reliably when users need them?

How to write one row

Principle

Usability

Current evidence

In my selected product, the checkout/search/sign-up flow becomes difficult when users need to...

Recommendation

Improve the flow by changing the label, grouping related fields, and adding clearer feedback after...

Impact

This reduces confusion because users can understand the next action before committing time or information.

This is a writing structure, not an answer. Students must replace every placeholder with their own product details.

Useful writing starters

Definition: “In UX, [principle] means...”

Current product: “In my case study, this succeeds/fails because...”

Recommendation: “One specific improvement would be...”

User impact: “This would help users by...”

T1 Q2

Consultation Techniques Before Design

Plan two ways to consult users or stakeholders before changing the product.

Technique option: User interviews

  1. Recruit participants who match the product’s target users.
  2. Prepare open-ended questions about goals, frustrations, and behaviour.
  3. Run short sessions and ask follow-up questions.
  4. Document exact notes, themes, and quotes.
  5. Translate findings into design requirements.

Technique option: Survey

  1. Define what you need to learn before writing questions.
  2. Mix demographic, behaviour, rating, and open-ended questions.
  3. Send to at least three relevant participants.
  4. Summarise patterns and contradictions.
  5. Use results to prioritise UX issues.
User quote Pain point Need

Question prompts students can adapt

  • “Tell me about the last time you used this product to complete...”
  • “Where did the process feel slow, confusing, or unnecessary?”
  • “What would make you trust this product more?”
  • “What would you do if this product did not work?”

Consultation technique menu

Interview Best when students need detailed stories, reasons, feelings, and examples.
Survey Best when students need responses from several people in the same format.
Observation Best when students can watch someone complete a task and note where they hesitate.
Stakeholder discussion Best when students need business goals, constraints, policies, or technical limits.

T1 Q3

Communication Techniques After Design

Explain how a proposed improvement would be communicated to audiences such as clients, developers, stakeholders, or users.

Method

Prototype walkthrough

Best for: clients, users, and stakeholders who need to see the changed experience.

Artefacts: clickable prototype, user flow, annotated screens, issue-to-solution notes.

Structure: problem recap, design goal, before/after flow, key screens, feedback request.

Method

Design handoff pack

Best for: developers or product teams who need to build the improvement.

Artefacts: wireframes, component notes, acceptance criteria, interaction states.

Structure: context, requirements, screen specs, behaviours, constraints, open questions.

For executives Lead with user impact, risk, cost, and expected outcomes.
For developers Lead with states, data, interactions, constraints, and edge cases.
For users Lead with clarity, task flow, what changed, and how to give feedback.

What “structured communication” can look like

Simple order: problem → evidence → proposed change → visual artefact → expected benefit → questions for feedback.

For developers: add states such as empty, loading, success, error, mobile, and desktop.

For clients: explain the user problem, why it matters, and what outcome the design should improve.

T2 Q1

Project Definition and Problem Framing

Define the social issue, user problem, target users, alternatives, success measures, SMART goal, and KPIs.

1. Project overview

Project name, type, social issue, and realistic context.

2. UX problem

What is currently happening? Why is it a problem? Who is affected?

3. Target users

Age, background, needs, behaviours, and context of use.

4. Justification

Why it matters, impact on users or society, risks if ignored.

5. Alternatives

Two direct solutions and two indirect alternatives, with strengths and gaps.

6. Success

What improves, what outcomes are expected, and how success is observed.

7. SMART goal

Specific, measurable, achievable, relevant, and time-bound.

8. KPIs

At least two measurable indicators with a measurement method.

Problem statement frame

“For [target users], [current situation] creates a problem because [user impact]. This matters because [social or practical consequence].”

A strong problem statement is specific. Avoid “the app is bad” or “people need help” without explaining the user, context, and impact.

S specific M measurable A achievable R relevant T time-bound

KPI menu

Task completion rateHow many users can finish the task without help?
Time on taskHow long does the task take before and after improvement?
Error rateHow often do users click the wrong thing, abandon the form, or misunderstand instructions?
User confidence ratingHow confident do users feel after using the solution?
Satisfaction scoreHow satisfied are users with the experience?

SMART goal frame

“Within [timeframe], improve [specific user task] for [target user group] by [measurable change], measured through [survey / task completion / error rate / observation].”

T2 Q2

User Research Design

Select a research method, define five suitable participants, and create at least ten purposeful questions.

Choose methodSurvey, interview, or mixed-method
Recruit usersFive people matching the target group
Write questionsEach question has a clear purpose
Collect dataUse real answers, not AI-generated responses

Good research questions

  • Ask about recent behaviour, not only opinion.
  • Use plain language.
  • Avoid leading users toward the “right” answer.
  • Connect to the problem and future design decisions.

Weak research questions

  • “Do you like my idea?”
  • “Wouldn’t it be better if the app was simpler?”
  • “Do you agree this is a serious problem?”
  • “What features should I build?” without context.

10-question planning pattern

1-2: Who they are 3-4: Current behaviour 5-6: Pain points 7-8: Needs and expectations 9: Existing alternatives 10: Success or preference

Research question bank

Background“How often do you use this type of service or product?”
Behaviour“Tell me about the last time you tried to complete this task.”
Pain point“Which part felt slow, unclear, stressful, or unnecessary?”
Workaround“What do you do when the product does not meet your needs?”
Expectation“What information or support would make this easier?”

Students must adapt questions to their project and add a purpose for every question.

T2 Q3

Research Findings and Synthesis

Document raw responses from five users, then identify patterns, problems, needs, and UX implications.

User 1User 2User 3User 4User 5
Repeated issue Shared need Contradiction Opportunity
UX implication What the design should consider next

Insight writing formula

“A pattern across the data is [finding]. This is supported by [number or quote evidence]. The UX implication is that the solution should [design consequence].”

How to find patterns without AI

Step 1: Highlight repeated words or ideas across participant answers.

Step 2: Group similar comments together, such as “confusing language,” “too many steps,” or “lack of trust.”

Step 3: Count how many users mentioned each issue.

Step 4: Write what the pattern means for design.

T3 Q1-Q2

Tool Selection and Initial Prompt

Explain what each tool is for, how they work together, and write an initial prompt grounded in Task 2.

Initial prompt structure

Context:
I am designing a UX solution for [target users] who experience [problem from Task 2].

Goal:
Create [interface / flow / prototype concept] that helps users [task or outcome].

Requirements:
- Include [key screens or features]
- Prioritise [needs from research]
- Use a tone and layout suitable for [user context]
- Avoid [risks or constraints]

Output:
Provide [wireframe description / screen list / user flow / prototype prompt / code brief].

Prompt improvement checklist

More context: name the user group, problem, setting, and constraints.

More specificity: ask for exact screens, components, flow, or interaction states.

More evaluation: explain what was useful, what was missing, and what you changed next.
